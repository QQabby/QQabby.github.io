1. sleep 和 wait 的区别？

  1. 两个方法都是 暂停线程，释放cpu 资源给其它线程 ；
  2. sleep 是 Thread 的方法 ， wait 是 Object 方法 ；
  3. sleep 使线程进入阻塞状态 ； wait 使线程进入等待状态 ，靠其它线程 notify 或 notifyAll 来改变状态 ；
  4. sleep 可以在任何地方使用，必须捕获异常 ； 而wait 必须在同步方法或同步代码块中执行（注意，wait()方法强制当前线程释放对象锁。这意味着在调用某对象的wait()方法之前，当前线程必须已经获得该对象的锁。），否则会抛出IllegalMonitorStateException:current ；
  5. 最重要的是：sleep 继续持有锁，wait 会释放锁 ，yield 停止当前线程，让同优先级或优先级高的线程先执行（但不会释放锁）；join 方法在某一个线程的执行过程中调用另一个线程执行，等到被调用的线程执行结束后，再继续执行当前线程。

2. Java 有哪些线程池？

一般来说有四种线程池

  1. CachedThreadPool

    可以根据需要创建新的线程的线程池，终止从缓存中移除那些已有60s未被使用的线程，因此，长时间保持空闲的线程池不会使用任何资源；

2.FixedThreadPool

创建了一个指定工作线程数量的线程池，每当提交一个任务就会创建一个线程，如果达到最大线程池初始的最大数，则将提交的任务存入池队列中 ；

  1. ScheduleThreadPool

    可安排在给定延迟后运行命令或定期的执行

4.SingleThreadPool

一个使用单个工作线程的线程池 ， 是 FixedThreadPool（1）的一种特殊情况 ；

3.1 线程池的工作流程

  1. 当来一个新的任务提交到线程池后，线程池首先判断核心线程池里的线程是否都在执行任务，如果不是，则创建一个新的工作线程来执行任务 ，如果是，则执行第二步：
  2. 线程池判断工作队列是否已经满了，如果工作队列没有满，则将新提交的任务存储在这个工作队列里等待；如果工作队列已经满了，则执行第三步：
  3. 线程池判断线程池里的线程是否都处于工作状态，如果没有，则创建一个新的工作线程来执行任务，如果满了，则交给 饱和策略 来处理这个任务。

饱和策略：

  1. 用户自定义拒绝策略（最常用的）： RejectedExecutionHandler ;
  2. CallerRunsPolicy ,阻塞入口等

3. 多线程间通信方式有哪些？

  1. synchronized 关键字这种方式来实现进程间的通信 ， 这种通信方式本质上是 “共享内存” 式的通信 ， 多个线程需要访问同一个共享变量，谁拿到了锁（获得了访问权限） ，谁就可以执行 ；

  2. wait/notify 机制

  3. 通过队列来实现线程的通信

    例如：LinkedBlockingQueue 来进行线程间交互 ， LinkedBlockingQueue 是一个单向链表，FIFO阻塞队列，访问与移除操作是在队头进行，添加操作是在队尾进行，通过共享一个队列的信息，实现生产者和消费者。

  4. Android 中还包括 Handler 消息通信机制，主线程与子线程之间的通信 ；

5. Thread 直接调用 run 方法会怎么样？ start 方法作用是什么？

Thread 直接调用 run 方法不会创建新的线程，run() 方法将作为当前调用线程本身常规的方法调用执行，并且不会发生多线程 ； run() 方法可以多次调用 。

Thread 调用 start() 方法是会创建一个新的线程；再者 start() 方法不能多次调用，否则抛出 IllegalThreadStateException ；

6.1 背景：jvm 运行时刻内存的分配

其中有一个内存区域就是 jvm 虚拟机栈 , 每一个线程 运行时都有一个线程栈，线程栈保存了线程运行时变量值信息。当线程访问某一个对象值得时候，首先通过对象得引用找到对应堆内存的变量的值，然后把堆内存变量的具体值load 到本线程本地内存中，建立一个变量副本，之后线程就不再和对象在堆内存变量值有任何关系，而是直接修改副本变量的值，在修改之后的某一个时刻（例如：线程退出之前），直接把线程变量副本的值写回到对象在堆中变量，这样堆中的对象值就发生了变化，如下图所示：

6.3 volatile 无法保证原子性

volatile 可以保证可见性，但无法保证原子性。

并发编程中的三个概念

    1. 原子性

      即一个操作或者多个操作，要么全部执行且执行过程不会被任何因素打断，要么就都不执行。我的理解是类似于数据库事务的操作，要么全部成功要么失败。

    示例：要从A账户向B转账100元，A账户有200元，B账户100元；包含两个操作：A账户减去100元，B账户增加100元，变为200元；

    假如A账户在转账100元，此时A账户已减去100元后被中止，又有另外一个线程对B账户进行取钱操作，取100，此时再继续执行操作，B账户+100，余额为100，而不是200，会出现问题。

    1. 可见性

      即当多个线程访问同一个变量时，一个线程修改了这个变量的值，其它线程能够立即看得到修改的值。

示例：

当 Thread1执行第一句话 i=10 这句话时，会把 i 的初始值加载到 CPU1 的高度缓存中，然后赋值为10 ，那么缓存中变为10了，但却没有立即写入到主存中；当Thread2 执行 j = i , 获取到 i 的值还是0 ，而不是10 ；

Java 内存模型只保证了基本读取和赋值是原子性操作，自增操作不属于原子性的；

Thread1 的修改，Thread2并没有立即看到 Thread1修改的值 ；没有保证可见性

      即程序执行的顺序按照代码的先后顺序执行。

6.4 volatile 能保证有序性吗？

volatile 能禁止指令重排，所以volatile 能在一定程度上保证有序性 ；

  1. 当程序执行到 volatile 变量的读操作或者写操作时，在其前面的操作肯定全部已经进行，且结果对后面的操作可见，在其后面的操作肯定还未进行；
  2. 在进行指令优化时，不能将对volatile变量访问的语句放在其后面执行，也不能把volatile变量后面的语句放到其前面执行。

6.5 volatile 的机制

加入 volatile 关键字和没有加入 volatile 关键字所生成的汇编代码发现，加入了 volatile 关键字时，会多出一个 lock 前缀指令。

lock 前缀指令实际上相当于一个内存屏障（也称内存栅栏），提供了3个功能 ：

  1. 确保指令重排不会把其后面的指令排到内存屏障之前的位置，也不会把前面的指令排到内存屏障的后面，即在执行到内存屏障这句指令时，在他前面的操作已经全部完成；
  2. 它会强制将对缓存的修改操作立即写入主存 ；
  3. 如果是写操作，它会导致其它CPU中对应的缓存无效；

7. 线程有哪几种状态？

一般认为线程有5种状态

1. New 
2. Runable 就绪准备状态
3. Running  运行状态
4. Block 阻塞状态（手动控制，例如加锁，等待，睡眠）
5. Dead  死亡状态
