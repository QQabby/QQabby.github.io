2. TCP/UDP 二者的区别

  1. TCP 是可靠的（经过三次握手，四次挥手），面向连接的协议 ，传输速度慢 ；
  2. UDP 是不可靠的，无连接的协议，协议简单，额外开销小，效率高，传输速度快，不能保证传输是否正确。

3. TCP和UDP对应的协议

TCP 对应的协议

  1. HTTP协议：从Web服务器传到超文本到本地浏览器的传送协议，默认使用80端口 ；
  2. FTP : 文件传输协议，默认使用21端口 ；
  3. SMTP : 邮件传送协议，用于发送邮件，默认使用25号端口 ；
  4. POP3 ：和 SMTP 对应，POP3 用于接收邮件，默认使用 110 端口；
  5. TEInet : 远程登录的端口 ，默认使用23端口

UDP 对应的协议

  1. DNS : 域名解析服务，将域名地址转化为 IP 地址 ，默认使用53端口 ；
  2. TFTP ： 简单文件传输协议，默认使用69端口 ；
  3. SNMP ： 简单网络管理协议，默认使用161端口，用来管理网络设备的 。

4. TCP 的报文段

TCP 是面向连接，可靠的传输协议，通过TCP报文进行传输，报文段分为头部和数据两部分

头部包含信息

原端口和目的端口 应用程序在客户端和服务端所对应的端口号
序号（seq）和确认序号（ack） TCP 可靠传输的关键部分，ack为1时才有效
数据偏移 头部长度不固定，最短20，最长60字节（补充后面）
保留 为将来定义新的用途保留，现在一般置为0
窗口 滑动窗口大小，用来告知发送端接收端的缓存大小，以此来控制发送端发送数据的速率，从而达到流量控制。

数据偏移：4位头部所能表示的最大值是 1111 ，转为十进制为 15 ，15*（32/8）= 60

5. TCP 三次握手，四次挥手

6. TCP 为什么是三次握手不是两次？

TCP 的三次握手最主要是防止已过期的连接再次传到被连接的主机，如果两次握手，有可能第一次握手客户端发送的报文段过了很久才到服务端，此时客户端已经不需要连接了，如果服务端建立了连接，就会造成服务器资源的浪费。

7. TCP流量控制：滑动窗口

  1. 前面在TCP 头部携带信息中提到过：窗口的概念； 所谓流量控制就是让发送方的发送速度不要太快，要让接收方来得及接收 ；
  2. 每次Client 返回的 ACK 都会告知 Server 当前 Client 窗口的大小，Server 会根据窗口的大小控制发送数据和向前滑动 ；

8. HTTP 协议包括哪些请求？

  1. get
  2. post
  3. update
  4. delete
  5. head : 跟 Get 方法相同 ，只不过服务器响应时不会返回消息体，可以用于获取请求中隐含的元信息，而不用传输实体本身，也经常用来测试超链接的有效性，可用性和最近的修改。
  6. options

9. Http 和 Socket 区别

  1. HTTP 是超文本传输协议 ， Socket 属于接口 ；
  2. Http 可能是基于Socket 的 ；
  3. Socket 可以维持一个长连接 ， HTTP 是请求响应式，通常 Socket 效率高 ；

10. Http 1.0/1.1/2.0 区别

  1. 1.1 相对于1.0：
    1. 支持长连接 : Http 1.1 支持长连接 和 请求的流水线处理 ， 在一个 TCP 连接上可以传送多个HTTP请求和响应，减少了建立和关闭连接的消耗和延迟，在HTTP 1.1 中默认开启了 Connection: keep-alive , 一定程序上弥补了 HTTP1.0 每次请求都要创建连接的缺点；
    2. 缓存处理 ：使用 if-Modified-Since , Expires 来做为缓存判断的标准 ， 1.1中引入了更多可供选择的缓存头来控制缓存策略（if-Modified-Since, if-Match, if-None-Match）
    3. 增加了 range 头域，支持断点续传；
    4. Host 头域 ： 在HTTP1.0 中认为每台服务器都绑定一个唯一的IP地址，因此，请求消息中的URL饼没有传递主机名（hostname） , 但随着虚拟主机技术的发展，一台服务器上可以存在多个虚拟主机，并且共享同一个IP地址，HTTP1.1 的请求消息和响应消息都应支持 Host 头域，如果没有Host 头域会报告一个错误（400 Bad Request）
  2. 2.0相对于1.x ：
    1. 支持多路复用 ；针对 HTTP高延迟的问题，多个请求stream 共享一个 tcp 连接的方式，解决了HOL Blocking 的问题，降低了延迟同时提高了宽带的利用率 ；可以设置优先级 ；
    2. 采用二进制解析 ： HTTP 1.x 的解析是基于文本。基于文本协议的格式解析存在天然缺陷，文本的表现形式有多样性，要做到健壮性需要考虑的场景很多，但是二进制则不同，只认0和1的组合，基于这种考虑Http2.0 解析决定采用二进制格式，实现方便且健壮；
    3. 首部压缩 ： header 头部压缩，选择合适的压缩算法可以减少包的大小和数量 ；
    4. 服务器推送 ： 采用了 SPDY 的网页，例如我的网页有一个 style,css 的请求，在客户端收到 style.css 数据的同时，服务器会将 style.js 文件推送给客户端，当客户端再次尝试获取 style.js 时就可以直接从缓存中获取到，不用再发请求了；

11. HTTPS

安全版本的HTTP ，基于 SSL 加密 ；

SSL 建立过程：

  1. 客户端给出（协议版本号，生成的随机数，客户端支持的加密方法）；
  2. 服务端确认双方使用的加密方法，并给出数字证书，以及服务器生成的随机数 ；
  3. 客户端确认数字证书有效，然后生成一个新的随机数，并使用数字证书中的公钥，加密这个随机数，发给服务器 ；
  4. 服务器使用自己的私钥，获取客户端发来额随机数 ；
  5. 客户端和服务端按照约定的加密方法，使用前面的三个随机数，生成 “对话密钥（session key）”, 用来加密接下来的整个对话过程 。

12. Cookie 和 Session 有什么区别

  1. 因为HTTP是无状态的，所以需要某种机制识别用户 ；
  2. Session 是在服务端保存的一个数据结构，用来跟踪用户的状态，这个数据可以保存在集群，数据库，文件中 ；
  3. Cookie 是客户端保存用户信息的一种机制 ，用来记录用户的一些信息，也是实现 Session 的一种方式

13. HTTP 状态码

  1. 消息 100 Continue 用来通知客户端它的部分请求已经被服务端接收，且仍未被拒绝 ；

  2. 成功 200区间段

  3. 重定向 300区间段
  4. 请求错误 400~
  5. 服务端错误 500~

14. 常见的请求头和响应头

请求头

  1. Accept : 可接受的响应内容类型 ，例如：Accept: text/plain
  2. Accept-Language : 可接受的响应内容的语言列表。 例如：Accept-Language:en-US
  3. Cache-Control : 是否使用缓存机制 。例如：Cache-Control : no-cache
  4. Cookie : Http 协议的Cookie 。 例如：Cookie：$Version=1,Skin=new ;
  5. Connection : 客户端想要优先使用的连接类型 。 例如： Connection: keep-alive ;
  6. Range : 断点续传 ；
  7. User-Agent : 浏览器的身份标识字符串 。 例如： User-Agent:Mozi…
  8. Date : 发送该消息的日期和时间

返回头

  1. Accept-Ranges : 服务器所支持的内容范围 。 例如：Accept-Ranges:bytes
  2. Date : 发送的日期和时间
  3. Cache-Control : 通知从服务端到客户端内的所有缓存机制，表示它们是否可以缓存这个对象及缓存有效时间
  4. Content-length : 响应消息体的长度，用8进制字节表示 。 例如： Content-Length: 348 ;
